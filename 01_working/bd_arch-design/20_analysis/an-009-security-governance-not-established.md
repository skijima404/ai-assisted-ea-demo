# Incident (Observed Behavior)
- 退職者がIDカードを返却しに来た際、社食システムを問題なく利用できてしまったケースが複数あったが、「頻度が低い」という理由で問題扱いされていない
- 人事システムでは退職処理が行われているにもかかわらず、社食システム側でアカウントが残存していることが **リスクとして認識されていない**
- 社食・人事・セキュリティ部門のいずれも、  
  「誰が退職者の利用停止を担保すべきか」が明確化されておらず、責任境界が曖昧
- 明確に“セキュリティインシデント予備軍”であるのに、  
  運用上は「例外的な事象」として扱われ続けている

# Conclusion
**退職者のアクセスが残存するという明確なセキュリティリスクがあるにもかかわらず、組織としてそのリスクを統治・監視・是正する仕組み（セキュリティガバナンス）が確立されていないため、問題が放置されている。**

# Assumptions / Hypotheses
- IDカード決済は“本人確認要素”として扱われているが、そのライフサイクルを管理する統制プロセスがない
- 人事は「退職処理は完了している」と認識し、社食は「人事側から明確な指示がない」と認識している構造的な責任断絶がある
- セキュリティ部門は社食アプリを“業務周辺系”として扱い、積極的なリスク監視を行っていない
- 「年間1〜2件」という低頻度が誤解を生み、インパクトの大きさが正しく評価されていない

# Reasoning
- 退職者がサービス利用できる＝不正アクセスの初期段階に該当する重大なセキュリティリスク  
  → これは“頻度”ではなく“インパクト”で評価すべき事象
- しかし社食運用チームは「低頻度だから問題ではない」と判断してしまっている  
  → ガバナンス不在により、重大リスクが“日常の例外扱い”として埋もれる
- HRDB と社食システム間で **公式なIDライフサイクル統制** が存在しない  
  → どのタイミングで誰が無効化を保証するかが定義されていない
- 結果として、退職者が利用できるというリスクが  
  **検知されない・報告されない・改善されない**  
  という状態が継続する
- この構造は、社食アプリ以外の周辺系業務にも波及するリスクがある

# Implications
- IDカードを認証要素として扱う全ての業務システムで、**退職情報の即時反映**を統制プロセスに含める必要がある
- 社食、人事、セキュリティ部門の間で、  
  - 退職時の無効化プロセス  
  - 例外時の承認フロー  
  - モニタリング（検知）  
  を含んだ統制を明確に定義する必要がある
- 社食システムが“周辺系”であることを理由にセキュリティ統制が緩い状態は、将来的により大きな事故を引き起こす可能性がある
- Transition Architecture では、  
  - ID無効化のイベントドリブン同期  
  - 退職者利用の検出ロジック（監査ログ）  
  を組み込むことが望ましい

# Observations
- 退職者の利用が「問題」と認識されていない
- 無効化プロセスがファイル連携に依存し、即時性を担保できていない
- ガバナンスの責任境界が曖昧で誰も対応を主担当として扱っていない
- 「頻度が低い」ことがリスクの深刻度評価を曇らせている
