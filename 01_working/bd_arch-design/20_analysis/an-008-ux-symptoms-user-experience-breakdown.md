# Incident (Observed Behavior)
- 社食利用金額が翌日にならないと反映されず、月末に予算超過へ気づけないケースが発生
- 社内コンビニで受け取ったレシート（割引後）とアプリ表示（割引前）が一致せず、問い合わせが頻発
- 新入社員が社食アプリを利用できるまで約1週間かかり、初期オンボーディングで不満が生じている
- 退職者が社食アプリを引き続き利用できてしまい、「データが正しくないのでは」という不信が生まれている
- 社食アプリの履歴画面は正確性が低く、「あまり見ていない」「どうせ正しくない」というユーザーの認識が形成されつつある

# Conclusion
**複数の領域で発生している整合性欠如・遅延・反映不備により、社食アプリ全体に対する“データ不信”がユーザー体験を大きく損ねている。これらは個別の不具合ではなく、構造的アーキテクチャ問題に起因する複合症状である。**

# Assumptions / Hypotheses
- ユーザー体験の問題はUI単体ではなく、データ反映の遅延・整合性欠如・責務分散などバックエンド構造由来
- CSS/フロント改善では解決できない「構造的UXの破損」が発生している
- 社食・コンビニ・人事のデータ境界が曖昧なまま統合されているため、正確な単一のソースが存在しない
- ユーザーが指摘する“信頼低下”は、複数ノード（an-002〜an-007）が共鳴して起きている

# Reasoning
- 金額反映が翌日になる（an-007 バッチ同期）  
  → 予算管理ができず「信頼できない」印象を与える
- レシートと画面の不一致（an-002, an-004）  
  → “アプリに書かれている金額そのものが誤っている”
- 新入社員の初期データ連携遅延（an-007）  
  → 「社食アプリってそういうもの」という諦め・低評価を与える
- 退職者が利用できる（an-005）  
  → 「重要情報が正しく処理されていない」という根源的不信につながる
- 本来分離されるべきドメイン（社食 vs コンビニ vs 人事）が混在（an-003, an-007）  
  → 表示金額の一貫性が初期から担保されない構造

結果として、UXの破損は **単一原因ではなく複数構造問題の合成結果** として現れている。

# Implications
- UX改善はUI/画面刷新ではなく、**データ鮮度・責務統一・反映プロセス改善**に着手しないと解決しない
- Baseline Architecture では「UX不信」を症状ノードとして明示し、複数原因へのリンクを可視化することで改善優先度を逆算できる
- Transition Architecture の計画においては、  
  - 金額の即時性（当日反映）  
  - 正確なレシート合計と割引反映  
  - IDライフサイクルの即時反映  
  の3点を UX KPI として扱うべき
- UX問題群を“単独の不具合”として扱うのではなく、**構造負債の症状群**として管理することが重要

# Observations
- 履歴画面の金額が信用されていない
- レシートとアプリの情報が一致しないことがある
- 金額反映が遅いため、社食アプリの利用が避けられる傾向がある
- 社食アプリは便利なはずだが、「信頼できなければ使われない」状態に近づきつつある
