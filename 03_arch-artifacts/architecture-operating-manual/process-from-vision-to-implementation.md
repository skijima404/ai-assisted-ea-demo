# アーキテクチャ成果物の定義と関係性（ビジョンから実装まで）

このドキュメントは、アーキテクチャビジョンを具体的な実装に落とし込むまでの、主要な成果物（Artifacts）の階層構造と定義をまとめたものです。

## 階層構造（土台から実装へ）

| 階層 | 成果物 (Artifact) | 定義 (Definition) | 関係性 (次の階層へのインプット) |
| :--- | :--- | :--- | :--- |
| **土台** | **アーキテクチャビジョン** | アーキテクチャが実現しなければならない**ビジネス価値**を定義したもの。 | アーキテクチャ原則を定めるための**目的**となる。 |
| **第2層** | **アーキテクチャ原則** | アーキテクチャビジョンを実現するための**設計や技術の基本方針**や**制約**。ここでは組織横断の原則と、プロジェクト固有の原則がある。 | リファレンスアーキテクチャの**再利用可能な標準構成**を導く。 |
| **第3層** | **リファレンスアーキテクチャ** | アーキテクチャ原則を実現できる**再利用可能な標準構成**。To Beアーキテクチャ策定時の**参照モデル**となる。 | To Beアーキテクチャの**全体像**と、**既存システム領域への適用**の基盤となる。 |
| **第4層** | **Targetアーキテクチャ** | **目標とするシステム全体像**。リファレンスアーキテクチャを元に**実施のシステム領域に落とし込んだ**もの。 | 設計、構築、運用の具体的な**活動の指針**となる。 |
| **最上位** | **実装** | 設計、構築、運用によりTo Beアーキテクチャを**具現化**したもの。 | — |

## TOGAF Tailoringについて

このモデルは **TOGAF（The Open Group Architecture Framework）** をベースにしながら、日本の大規模企業における実務的な合意形成をスムーズにするため、意図的に2点のTailoringを加えています。

### Tailoringの意図

1. **ビジョンをビジネス側に強く置く**  
   「何の目的で構造を変えるのか」という合意を早期に得やすくするため、アーキテクチャビジョンに明確に**ビジネス価値の定義**を求めています。

2. **アーキテクチャ原則の役割分担を明確化**  
   アーキテクチャ原則の上流（組織横断の原則）をビジネス+技術、下流（プロジェクト固有の原則）を技術側が担う形にすることで、ビジネスと技術の役割と橋渡しを明確化しています。

### 具体的なTailoringポイント

#### 1. アーキテクチャビジョンに「ビジネス価値を定義したもの」というルールを追加

**TOGAF標準**: アーキテクチャビジョンは主に Phase A で定義されますが、ビジネス価値の明示的な定義は必須要件ではありません。

**本モデル**: アーキテクチャビジョンには**必ず**ビジネス価値を定義することを求めています。これにより：
- ステークホルダー（特に経営層）との合意形成が早まる
- 技術的な議論が「手段」に終始せず、「目的」から始められる
- アーキテクチャ原則の導出に明確な根拠ができる

#### 2. アーキテクチャ原則にプロジェクト固有原則を含める

**TOGAF標準**: アーキテクチャ原則は主に組織全体の原則（Enterprise Principles）として扱われます。

**本モデル**: 組織横断の原則とプロジェクト固有の原則の両方を含めています。これにより：
- 組織全体の標準と、個別プロジェクトの特性を両立できる
- プロジェクト固有の技術的判断を、原則レベルで明文化・承認できる
- リファレンスアーキテクチャとの適用関係が明確になる

### 効果

このTailoringにより、以下の効果を実現しています：

1. **合意形成のスピード向上**  
   ビジネス価値が明確なため、経営層やステークホルダーとの合意が早期に得られる

2. **責任範囲の明確化**  
   ビジネスサイドと技術サイドの役割分担が明確で、建設的な議論が可能

3. **再現性の確保**  
   組織横断の原則とプロジェクト固有の原則を分離することで、次のプロジェクトへの知見展開が容易

4. **トレーサビリティの向上**  
   ビジョン → 原則 → リファレンス → Target → 実装の流れが明確で、各成果物の根拠が追跡可能
